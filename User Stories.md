## Ниже представлены пользовательские сценарии, приведённые в соответствие с реализацией (код, тесты, API) и лучшими практиками (чёткая структура, полные альтернативы, явные статусы).

### 1. Регистрация пользователя
   Актор: Новый пользователь

   Предусловие: Пользователь не аутентифицирован, `email` не занят.

   Основной сценарий (успех):

    Пользователь отправляет POST‑запрос на /api/v1/auth/register с JSON:

    name (обязательно, 2–100 символов)

    email (обязательно, валидный email)

    password (обязательно, минимум 6 символов)

    phone (опционально)

    deliveryAddress (опционально)

  Система проверяет уникальность `email`.

  Система создаёт запись в таблице `users` с ролью USER. Пароль хэшируется (BCrypt).

  Система автоматически создаёт для пользователя:

    счёт в billing‑сервисе (баланс = 0)

    пустую корзину в order‑сервисе

  Система генерирует JWT‑токен (срок действия задаётся в security.jwt.expiration).

  Система возвращает HTTP 201 Created с телом:

    `json
    {
    "token": "JWT_TOKEN",
    "type": "Bearer",
    "user": { "id": 123, "name": "...", "email": "...", ... }
    }`

  Альтернативы и исключения:

  `Email` уже существует → 409 Conflict, код EMAIL_ALREADY_EXISTS.

  Невалидные входные данные (пропущены обязательные поля, `email` невалиден) → 400 Bad Request, код VALIDATION_ERROR.

  Внутренняя ошибка → 500 Internal Server Error.

  Постусловие: Пользователь аутентифицирован, может пополнять баланс, добавлять блюда в корзину, оформлять заказы.

### 2. Пополнение баланса
   Актор: Зарегистрированный пользователь (аутентифицирован)
   
   Предусловие: Пользователь существует в billing‑сервисе.

   Основной сценарий:

   - Пользователь отправляет POST‑запрос на /api/v1/accounts/{userId}/deposit с телом:

   `json
   { "amount": 2000.00 }`

    Сумма должна быть положительной (минимальная сумма 0.01).

    Система проверяет существование счёта.

    Система увеличивает баланс на указанную сумму.

   Система возвращает HTTP 200 OK с телом:

   `json
   { "id": 1, "userId": 123, "balance": 2000.00 }`

   Альтернативы и исключения:

    Сумма ≤ 0 → 400 Bad Request (валидация).

    Пользователь не найден → 404 Not Found, код ACCOUNT_NOT_FOUND.

    Ошибка сервера → 500 Internal Server Error.

   Постусловие: 
    
    Баланс пользователя увеличен.

### 3. Работа с корзиной
   
#### 3.1 Добавление товара в корзину
   Актор: Зарегистрированный пользователь

   Предусловие: Корзина пользователя существует (создаётся автоматически при регистрации).

   Основной сценарий:

   - Пользователь отправляет POST‑запрос на /api/v1/carts/{userId}/items с телом:

      `json
      { "productId": 1, "quantity": 1 }`

   - Система находит корзину пользователя.

   - Если товар уже есть в корзине – увеличивает его количество; иначе – добавляет новую позицию.

   - Система возвращает HTTP 200 OK с актуальным содержимым корзины (список позиций: id, productId, quantity).

   Альтернативы и исключения:

    Корзина не найдена → 404 Not Found (код CART_NOT_FOUND).

    Некорректные данные (productId=null, quantity≤0) → 400 Bad Request.

#### 3.2 Удаление позиции из корзины

   Актор: Зарегистрированный пользователь

   Предусловие: Корзина существует, позиция с указанным itemId принадлежит этой корзине.

   Основной сценарий:

   - Пользователь отправляет DELETE‑запрос на /api/v1/carts/{userId}/items/{itemId}.

    Система удаляет указанную позицию из корзины.

    Возвращает HTTP 200 OK с обновлённой корзиной.

   Альтернативы: Если позиция не принадлежит корзине → 400 Bad Request.

#### 3.3 Замена содержимого корзины
   Актор: Зарегистрированный пользователь

   Используется в тестах для атомарной установки корзины.

   - Пользователь отправляет PUT‑запрос на /api/v1/carts/{userId}/items с телом:


      `json
      { "items": [ { "productId": 1, "quantity": 1 }, ... ] }`


    Система полностью заменяет содержимое корзины на указанный список.

    Возвращает HTTP 200 OK с новой корзиной.

#### 3.4 Просмотр корзины

  Актор: Зарегистрированный пользователь

  Предусловие: Корзина существует.

   Основной сценарий:

   - Пользователь отправляет GET‑запрос на /api/v1/carts/{userId}.

    Система возвращает список позиций с productId и quantity.

   Примечание: Цены и названия в текущей версии не возвращаются.

   Альтернативы: Корзина пуста → возвращается пустой список.
   
   Постусловие: Пользователь видит содержимое корзины.

### 4. Оформление заказа (сложный сценарий с ожиданием готовки)
   
   Актор: Зарегистрированный пользователь
   
   Предусловия:

    Корзина не пуста.

    Пользователь аутентифицирован.

  Основной сценарий (без товаров на готовку):

  - Пользователь отправляет POST‑запрос на /api/v1/carts/{userId}/checkout с телом:


    `json
    {
    "userId": 123,
    "deliveryTime": "2026-04-18 21:00:00",   // опционально
    "acceptWaiting": true                    // обязательно
    }`
  Система проверяет:

  - наличие всех товаров на складе (доступны сейчас);

  - достаточность баланса (через billing);

  - наличие свободного слота доставки (если deliveryTime не указан – берётся ближайший).

  Если все проверки пройдены, система:

  - списывает средства;

  - резервирует товары;

  - резервирует слот доставки;

  - создаёт заказ со статусом CONFIRMED;

  - очищает корзину;

  - отправляет уведомление (но не ждёт ответа).

  - Возвращает HTTP 200 OK с объектом заказа (id, сумма, статус CONFIRMED, время доставки и т.д.).

  Сценарий с блюдами, требующими готовки:

  - Пользователь отправляет запрос на checkout с acceptWaiting: true.

  - Система обнаруживает, что хотя бы один товар имеет preparationTimeMinutes > 0 и недоступен сейчас.

  - Система создаёт заказ со статусом PENDING, не списывая средства и не резервируя слот.

  - Система отправляет уведомление с просьбой подтвердить заказ (в теле – информация о времени готовки).

  - Пользователь подтверждает заказ через POST /api/v1/orders/{orderId}/confirm?accept=true.

  Система:

  - проверяет баланс и наличие слотов доставки с учётом времени готовки (текущее время + preparationTimeMinutes + minDeliveryTime);

  - списывает средства;

  - резервирует товары;

  - резервирует слот доставки;

  - меняет статус заказа на CONFIRMED;

  - очищает корзину.

  Если пользователь отказывается (accept=false), заказ переходит в статус CANCELLED, корзина не очищается.

  Альтернативы и исключения:

    Корзина пуста → 400 Bad Request (Cart is empty).

    Недостаточно средств → 402 Payment Required (код INSUFFICIENT_FUNDS).

    Нет свободного слота на указанное время (или после времени готовки) → возвращается 409 Conflict с телом, содержащим ближайшие доступные слоты.

    Пользователь не согласен ждать (acceptWaiting=false), но есть товары на готовку → 400 Bad Request с сообщением о необходимости подтверждения.

    Невалидное время доставки (в прошлом) → 400 Bad Request.

  Постусловие (успех): Заказ в статусе CONFIRMED, баланс уменьшен, корзина пуста.
  
  Постусловие (PENDING): Заказ ожидает подтверждения, корзина не очищена.

### 5. Подтверждение заказа (для PENDING)

   Актор: Зарегистрированный пользователь

   Предусловие: Существует заказ в статусе PENDING, принадлежащий пользователю.

   Основной сценарий:

   - Пользователь отправляет POST /api/v1/orders/{orderId}/confirm?accept=true.

   - Система проверяет баланс и доступность слота доставки с учётом времени готовки.

   - При успехе списывает средства, резервирует товары и слот, меняет статус на CONFIRMED, очищает корзину.

   - Возвращает HTTP 200 OK с обновлённым заказом (статус CONFIRMED).

   Альтернативы:

   accept=false → заказ переходит в CANCELLED, корзина не очищается, возвращается статус CANCELLED.

   Недостаточно средств → заказ в FAILED, возвращается ошибка 402.

   Нет слота доставки → возвращается список ближайших слотов (код 409).

   Заказ не в статусе PENDING → 400 Bad Request.

### 6. Доставка заказа (отметка курьером)

   Актор: Курьер (система доставки или внутренний вызов)
   
   Предусловие: Заказ в статусе CONFIRMED (или READY_FOR_DELIVERY – в текущей реализации проверяется CONFIRMED или READY_FOR_DELIVERY).

   Основной сценарий:

   - Курьер отправляет POST /api/v1/orders/{orderId}/deliver.

   Система:

   - меняет статус заказа на DELIVERED;

   - освобождает слот доставки (вызов delivery‑сервиса);

   - коммитит резервацию товаров (уменьшает фактические остатки на складе);

   - отправляет уведомление пользователю.

   - Возвращает HTTP 200 OK без тела (или с пустым телом).

   Альтернативы:

   Заказ не в статусе CONFIRMED или READY_FOR_DELIVERY → 400 Bad Request с сообщением "Order cannot be delivered, current status: ...".

   Ошибка при коммите товаров (например, недостаточно остатков) → 500 Internal Server Error (заказ остаётся в DELIVERED, но склад может быть несогласован; в идеале нужна компенсация, но в текущей версии просто ошибка).

   Постусловие: Заказ в статусе DELIVERED.

### 7. Подтверждение получения заказа пользователем
   
   Актор: Зарегистрированный пользователь
   
   Предусловие: Заказ в статусе DELIVERED и принадлежит пользователю.

   Основной сценарий:

   - Пользователь отправляет POST /api/v1/orders/{orderId}/confirm-receipt.

   Система проверяет, что заказ существует и имеет статус DELIVERED.

   Система меняет статус на COMPLETED.

   Возвращает HTTP 200 OK.

   Альтернативы и исключения:

   Заказ не в статусе DELIVERED → 400 Bad Request ("Order is not in delivered state").

   Заказ не найден → 404 Not Found.

   Заказ не принадлежит пользователю → 403 Forbidden (в текущей реализации проверка не выполняется – это потенциальный баг, но сценарий должен упоминать ожидаемое поведение).

   Постусловие: Заказ завершён, дальнейшие изменения невозможны.

### 8. Идемпотентное создание заказа (прямой заказ без корзины)
   
   Актор: Зарегистрированный пользователь (или внешняя система)
   
   Предусловие: Клиент генерирует уникальный ключ идемпотентности (например, UUID).

   Основной сценарий:

   - Пользователь отправляет POST /api/v1/orders с заголовком Idempotency-Key: <уникальный ключ> и телом, содержащим userId, items, deliveryTime и т.д.

   Система ищет запись с таким ключом в таблице idempotency_records.

   Если найдена – возвращает ранее созданный заказ (без повторного выполнения).

   Если нет – создаёт новый заказ (аналогично сценарию из корзины), сохраняет связку ключ → заказ.

   Возвращает созданный заказ.

   Альтернативы:

   Отсутствие заголовка Idempotency-Key → 400 Bad Request (MISSING_IDEMPOTENCY_KEY).

   Повторный запрос с тем же ключом → возвращается тот же заказ (статус может быть любым).
